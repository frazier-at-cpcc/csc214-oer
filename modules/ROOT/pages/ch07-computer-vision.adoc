== 7.1 Introduction: Teaching Machines to See

In xref:ch06-nlp-and-speech.adoc[Chapter 6], you learned how Azure AI services extract structured information from unstructured text and speech. You applied pre-built analytics, built custom classification models, and composed multi-service pipelines. This chapter applies the same architectural thinking to a different kind of unstructured data: images.

Consider this scenario. A wildlife conservation organization -- the Piedmont Wildlife Trust -- manages a network of 200 trail cameras deployed across protected habitats in the Carolina Piedmont region. Every week, these cameras capture roughly 50,000 images. Some images show white-tailed deer, black bears, coyotes, or red foxes. Others show birds, reptiles, or the occasional domestic animal that has wandered into the preserve. Many images show nothing but waving grass or swaying branches triggered by motion sensors. Each image needs to be reviewed, the species identified, and the sighting logged in the Trust's biodiversity database.

But species identification is only part of the problem. Field researchers attach physical tags to certain animals -- small plastic ear tags with handwritten identification codes. When the trail cameras capture a tagged animal, the researchers need to read the tag number from the image and match it to their records. Some researchers also leave handwritten field notes on waterproof cards propped in front of cameras as status updates. These notes need to be transcribed into the database as well.

The Trust's conservation director wants to automate this process. She envisions a system that can sort through the 50,000 weekly images, discard the empty ones, identify the species in each remaining image, detect and read any visible tags or field notes, and flag unusual sightings for researcher review. She also wants researchers to be able to upload a photo from their phone and ask the system questions like "What species is this?" or "Is this animal showing signs of mange?" -- questions that require flexible visual reasoning rather than rigid classification.

This is not a single-service problem. It requires image analysis to tag and describe each photo. It requires optical character recognition to read text from tags and field notes. It requires custom image classification to distinguish between species that pre-built models cannot differentiate -- say, a gray fox from a red fox, or a Cooper's hawk from a sharp-shinned hawk. It requires face detection principles to understand how detection differs from identification. And it requires vision-enabled generative AI to handle the open-ended questions that researchers ask in the field.

This chapter walks through each service individually, then helps you build a decision framework for choosing the right service for each task. Here is the roadmap:

* **Section 7.2** covers Azure AI Vision's Image Analysis capabilities: tagging, object detection, captions, and smart cropping.
* **Section 7.3** introduces the Read API for optical character recognition.
* **Section 7.4** explores Custom Vision for training domain-specific image classification and object detection models.
* **Section 7.5** covers face detection and its responsible AI boundaries.
* **Section 7.6** introduces vision-enabled generative AI -- sending images to chat models and generating images from text.
* **Section 7.7** provides a decision framework for choosing among these services.
* **Sections 7.8 and 7.9** connect the chapter to your hands-on labs and the written assessment.

== 7.2 Image Analysis

Azure AI Vision's **Image Analysis** service examines an image and returns structured information about its contents. You send an image -- either as a URL or as raw binary data -- and the service returns tags, objects with bounding boxes, captions, and other metadata. No training is required. The service works immediately with a pre-built model trained on millions of images across thousands of categories.

For the Piedmont Wildlife Trust, Image Analysis handles the first pass through the weekly image haul. It can tag each image with descriptive labels ("animal," "deer," "forest," "outdoor"), detect distinct objects within the frame, and generate natural-language captions that summarize what the image shows. This first pass filters out the empty images (tagged only with "grass" or "tree" and no animal-related tags) and provides a rough species classification that researchers can refine.

You provision an Azure AI Vision resource in the Azure portal, obtain your endpoint URL and API key, and start making API calls. A single call can request multiple analysis features simultaneously -- tags, objects, captions, and smart crop suggestions in one round trip.

=== 7.2.1 Image Tagging

**Image tagging** returns a list of descriptive words and phrases that apply to the image. Each tag comes with a confidence score between 0 and 1 indicating how certain the model is about the tag's relevance. An image of a deer standing in a meadow might return tags like "deer" (0.97), "animal" (0.99), "mammal" (0.95), "outdoor" (0.98), "grass" (0.92), "wildlife" (0.89), and "nature" (0.87).

Tags describe the image at multiple levels of abstraction. You get both specific labels ("deer") and general categories ("animal," "mammal"), along with contextual tags about the scene ("outdoor," "grass," "forest"). This hierarchical tagging is useful for building search indexes -- a researcher searching for "mammal" should find images of deer, bears, and foxes alike.

The pre-built tagging model recognizes thousands of objects, scenes, and concepts. However, it operates at a general level. It will reliably distinguish a deer from a bear, but it may not distinguish a white-tailed deer from a mule deer. For that level of specificity, you will need Custom Vision (Section 7.4).

=== 7.2.2 Object Detection with Bounding Boxes

**Object detection** goes beyond tagging by identifying the location of each recognized object within the image. For each detected object, the service returns a label, a confidence score, and a **bounding box** -- a rectangle defined by its top-left corner coordinates, width, and height. The bounding box coordinates are expressed in pixels relative to the original image dimensions.

An image containing two deer and a bird might return three bounding boxes: two labeled "deer" and one labeled "bird," each outlining the specific region of the image where that animal appears. This spatial information is essential when multiple objects share a single frame -- which is common with trail cameras that capture herds or mixed-species gatherings.

For the Wildlife Trust, object detection enables accurate counting. Instead of knowing only that an image "contains deer," the system can report "three deer detected at positions (x1, y1), (x2, y2), (x3, y3)." This data feeds directly into population density estimates and movement tracking analyses.

=== 7.2.3 Image Description and Caption Generation

**Caption generation** produces a natural-language sentence describing the image's content. The Azure AI Vision service generates captions using a model that understands both the objects in the image and their relationships. An image of a bear near a stream might produce the caption "A black bear standing near a stream in a forested area."

The service can generate both a single primary caption and a set of dense captions. **Dense captions** describe multiple regions of the image independently, providing more detailed coverage. For a complex trail camera image showing a deer in the foreground and birds in the background, dense captions might include "A deer standing in tall grass," "Small birds perched on a branch," and "A forested hillside under cloudy sky."

Captions serve multiple purposes for the Trust. They provide human-readable descriptions for the biodiversity database, they enable natural-language search ("find images of bears near water"), and they improve accessibility for visually impaired researchers reviewing image catalogs.

=== 7.2.4 Smart Cropping and Thumbnails

**Smart cropping** generates thumbnails that focus on the most important region of an image. Rather than simply scaling the image down or cropping from the center, smart cropping uses the analysis model to identify the region of interest -- typically the main subject -- and crops around it.

For trail camera images, smart cropping ensures that thumbnails in the image review interface actually show the animal rather than a tiny creature in the corner of a large landscape frame. When researchers browse through hundreds of thumbnail previews, a well-cropped thumbnail showing a clear view of the animal is far more useful than a shrunken full-frame image where the animal occupies only a small fraction of the pixels.

=== 7.2.5 Confidence Scores

Every result from Image Analysis includes a **confidence score** -- a decimal value between 0 and 1. A tag with a confidence score of 0.97 means the model is highly confident in that label. A tag with a score of 0.45 means the model is uncertain.

You should use confidence scores to filter results. In production, you will set a **confidence threshold** -- a minimum score that a result must meet to be accepted. For the Wildlife Trust's initial image filtering, a threshold of 0.70 on animal-related tags might be appropriate: images that score below this threshold on all animal tags get flagged as "likely empty" rather than automatically discarded, preserving the ability for a human to review edge cases.

Confidence scores also indicate when you need a more specialized model. If the Image Analysis service consistently returns "animal" with high confidence but "fox" with low confidence for fox images, that is a signal that the general model struggles with your specific classification needs -- and a cue to train a Custom Vision model.

=== 7.2.6 Use Cases

Image Analysis is broadly useful across many industries. Retail organizations use it to automatically tag product images for e-commerce catalogs. Manufacturing companies use object detection to identify defects on production lines. Media companies use caption generation to produce alt text for images at scale. Real estate platforms use smart cropping to generate consistent property photo thumbnails. Insurance companies use image tagging to classify claim photos by damage type.

The common thread is that these are general-purpose visual understanding tasks where the pre-built model's broad training covers the needed categories. When your task requires categories that the general model does not understand, you move to Custom Vision.

The following code example demonstrates a complete Image Analysis API call that requests tags, objects, and a caption for a trail camera image.

[source,python]
----
import os
from azure.ai.vision.imageanalysis import ImageAnalysisClient
from azure.ai.vision.imageanalysis.models import VisualFeatures
from azure.core.credentials import AzureKeyCredential

# Authenticate with your Azure AI Vision resource
endpoint = os.environ["AZURE_VISION_ENDPOINT"]
key = os.environ["AZURE_VISION_KEY"]
client = ImageAnalysisClient(endpoint=endpoint, credential=AzureKeyCredential(key))

# Analyze an image from a URL
image_url = "https://example.com/trail-cam/IMG_20260301_0742.jpg"

result = client.analyze_from_url(
    image_url=image_url,
    visual_features=[
        VisualFeatures.TAGS,
        VisualFeatures.OBJECTS,
        VisualFeatures.CAPTION,
        VisualFeatures.DENSE_CAPTIONS,
        VisualFeatures.SMART_CROPS
    ],
    smart_crops_aspect_ratios=[1.0, 1.5]  # Square and 3:2 thumbnails
)

# Display the caption
if result.caption is not None:
    print(f"Caption: '{result.caption.text}' "
          f"(confidence: {result.caption.confidence:.2f})")

# Display tags above the confidence threshold
threshold = 0.70
print(f"\nTags (confidence >= {threshold}):")
if result.tags is not None:
    for tag in result.tags.list:
        if tag.confidence >= threshold:
            print(f"  {tag.name}: {tag.confidence:.2f}")

# Display detected objects with bounding boxes
print("\nDetected objects:")
if result.objects is not None:
    for obj in result.objects.list:
        box = obj.bounding_box
        print(f"  {obj.tags[0].name} "
              f"(confidence: {obj.tags[0].confidence:.2f}) "
              f"at [{box.x}, {box.y}, {box.width}, {box.height}]")
----

[TIP]
====
Image Analysis is a generalist -- it recognizes thousands of everyday objects and scenes, but it does not understand domain-specific categories. If you need to distinguish between closely related species, product variants, or custom classification categories, use Custom Vision (Section 7.4) to train a model on your own labeled data.
====

== 7.3 OCR and the Read API

The Azure AI Vision **Read API** extracts text from images and documents. Whether the text is printed on a sign, typed on a form, or handwritten on a field note card, the Read API detects the text regions, recognizes the characters, and returns the extracted text as structured data. For the Piedmont Wildlife Trust, the Read API is the service that reads animal tag numbers from trail camera images and transcribes handwritten field notes left by researchers.

The Read API is optimized for real-world text recognition -- text that appears in photographs under imperfect conditions. Unlike document scanners that work with clean, high-contrast scans, the Read API handles text at odd angles, text on curved surfaces, text partially obscured by shadows, and text captured by cameras at varying distances. This robustness makes it suitable for the Trust's trail camera images, where tag numbers might be partially visible, at an angle, or in low-light conditions.

=== 7.3.1 Printed Text Extraction

**Printed text extraction** handles machine-generated text: typed labels, printed signs, digital displays, and other text rendered in standard fonts. The Read API detects lines and words within the image, returning each word's text content, its bounding polygon (the coordinates of its corners in the image), and a confidence score.

For the Wildlife Trust, printed text appears on manufactured animal tags, equipment labels, and trail camera date stamps. The date stamp burned into each trail camera image -- typically showing the date, time, and camera identifier -- can be extracted automatically and used to populate the database record for each image without manual entry.

=== 7.3.2 Handwritten Text Extraction

**Handwritten text extraction** recognizes text written by hand. Handwriting varies enormously between individuals in letter formation, spacing, slant, and size, making it significantly more challenging than printed text recognition. The Read API uses deep learning models trained on diverse handwriting samples to handle this variability.

The Trust's field researchers leave handwritten notes on waterproof cards propped in front of trail cameras. These notes might read "Camera 14 -- battery replaced 2/28 -- salt lick moved 50m north" or "Possible den site 30m east -- DO NOT DISTURB." The Read API extracts this text so it can be logged in the database alongside the camera's automated image captures, giving researchers a complete record of field activities without requiring them to carry electronic devices into remote locations.

Handwritten text recognition accuracy depends on handwriting legibility. Neat, well-spaced block lettering produces much better results than hurried cursive. When deploying a system that relies on handwritten text extraction, you should establish guidelines for field note formatting and validate extraction results against a quality threshold.

=== 7.3.3 Multi-Language Support

The Read API supports text recognition in dozens of languages, including Latin-script languages (English, Spanish, French, German, and many others), as well as Chinese, Japanese, Korean, Arabic, and Cyrillic scripts. The service automatically detects the language of the text in the image -- you do not need to specify it in advance.

For the Wildlife Trust, multi-language support is relevant when collaborating with international conservation organizations. A partner organization in Costa Rica might share trail camera images with Spanish-language field notes, and the Read API will extract that text without requiring any configuration changes.

=== 7.3.4 Document Layout Awareness

The Read API goes beyond simple character recognition by understanding **document layout** -- the spatial organization of text on a page or in an image. It identifies paragraphs, lines, words, and their reading order. For multi-column documents, the API determines which text belongs to which column and returns the text in the correct reading sequence rather than simply scanning left to right across the entire width.

This layout awareness matters when the Trust processes multi-section field notes or printed research summaries. A field note card with two columns -- one for observations and one for measurements -- will be read correctly as two separate streams of text rather than a jumbled alternation between columns.

=== 7.3.5 Accuracy Factors

Several factors affect OCR accuracy:

* **Image quality.** Higher resolution produces better results. Blurry, low-contrast, or heavily compressed images reduce accuracy.
* **Lighting conditions.** Even illumination is ideal. Deep shadows, glare, and extreme backlighting degrade recognition.
* **Text orientation.** The Read API handles rotated and skewed text, but extreme angles reduce accuracy.
* **Font and handwriting style.** Standard fonts are recognized with near-perfect accuracy. Decorative fonts and poor handwriting are less reliable.
* **Background complexity.** Text on a plain background is easier to detect than text overlaid on a busy photographic background.

For trail camera images, several of these factors work against you simultaneously: low resolution, poor lighting (many captures occur at dawn or dusk), and busy natural backgrounds. Setting realistic accuracy expectations and building human review into your workflow for low-confidence extractions is essential.

=== 7.3.6 Read API vs. Document Intelligence OCR

Azure offers two services that extract text from documents: the Read API (part of Azure AI Vision) and Azure AI Document Intelligence (covered in xref:ch08-document-intelligence-and-knowledge-mining.adoc[Chapter 8]). The Read API extracts raw text -- it tells you what words appear in the image and where they are located. Document Intelligence goes further by understanding document structure: it identifies fields, tables, key-value pairs, and document types. It can extract a name from a specific field on a form, read rows and columns from a table, and classify a document as an invoice, receipt, or contract.

The rule of thumb: use the Read API when you need raw text extraction from images and photographs. Use Document Intelligence when you need structured data extraction from forms, invoices, receipts, and other business documents. The Trust's trail camera tags and handwritten field notes are Read API tasks -- they contain unstructured text that needs to be captured. If the Trust later needed to process expense reports or equipment purchase orders, Document Intelligence would be the better choice.

[NOTE]
====
The Read API extracts raw text from images and documents. If you need to extract structured form data -- key-value pairs, table rows, or specific fields from known document types -- use Azure AI Document Intelligence, which is covered in xref:ch08-document-intelligence-and-knowledge-mining.adoc[Chapter 8].
====

The following code example demonstrates using the Read API to extract text from a photograph of a whiteboard containing research notes.

[source,python]
----
import os
from azure.ai.vision.imageanalysis import ImageAnalysisClient
from azure.ai.vision.imageanalysis.models import VisualFeatures
from azure.core.credentials import AzureKeyCredential

# Authenticate with your Azure AI Vision resource
endpoint = os.environ["AZURE_VISION_ENDPOINT"]
key = os.environ["AZURE_VISION_KEY"]
client = ImageAnalysisClient(endpoint=endpoint, credential=AzureKeyCredential(key))

# Read text from a whiteboard photo
image_url = "https://example.com/fieldwork/whiteboard-notes-20260301.jpg"

result = client.analyze_from_url(
    image_url=image_url,
    visual_features=[VisualFeatures.READ]
)

# Display extracted text organized by blocks and lines
if result.read is not None:
    for block in result.read.blocks:
        for line in block.lines:
            # Each line includes text and a bounding polygon
            print(f"Line: '{line.text}'")
            print(f"  Bounding polygon: {line.bounding_polygon}")

            # Each word includes text, confidence, and bounding polygon
            for word in line.words:
                print(f"  Word: '{word.text}' "
                      f"(confidence: {word.confidence:.2f})")
        print()  # Blank line between blocks
----

== 7.4 Custom Vision: Training Domain-Specific Models

The pre-built Image Analysis service from Section 7.2 recognizes thousands of general categories. It will tell you that an image contains a "bird" or a "deer." But the Piedmont Wildlife Trust needs more than that. They need to know whether the bird is a Cooper's hawk or a sharp-shinned hawk, whether the deer is a white-tailed deer or a mule deer, and whether the fox is a red fox or a gray fox. These distinctions matter for biodiversity tracking, population studies, and conservation planning -- and the general-purpose model does not make them.

This is where **Custom Vision** comes in. Custom Vision is an Azure AI service that lets you train your own image classification and object detection models using your own labeled data. You upload images, tag them with your custom categories, train a model, evaluate its performance, and deploy it to a prediction endpoint. The result is a model that understands your specific domain.

=== 7.4.1 When Pre-Built Image Analysis Isn't Enough

The decision to build a custom model should follow the same principle you learned in xref:ch06-nlp-and-speech.adoc[Chapter 6] for text classification: start with pre-built capabilities, evaluate their accuracy on your data, and train custom models only to fill the gaps.

Run your images through the pre-built Image Analysis service first. If it correctly classifies your images at the granularity you need, you are done. If it consistently returns "bird" when you need "Cooper's hawk," or "fox" when you need "red fox," then you have identified a gap that Custom Vision can fill.

Custom models require effort to build and maintain. You need labeled training data, a training cycle, ongoing evaluation, and periodic retraining as your data evolves. Do not invest that effort unless the pre-built model genuinely falls short of your requirements.

=== 7.4.2 Image Classification

**Image classification** assigns one or more category labels to an entire image. You define the categories (your "classes"), upload labeled training images for each category, and train a model that learns to distinguish between them.

Custom Vision supports two classification modes:

* **Multiclass classification** assigns exactly one label per image. Use this when categories are mutually exclusive -- an image shows either a red fox or a gray fox, but not both.
* **Multilabel classification** assigns one or more labels per image. Use this when categories can overlap -- an image might show both a deer and a bird in the same frame.

For the Wildlife Trust's species identification task, multiclass classification is the right choice for single-animal images (each image contains one species), while multilabel classification handles the multi-species frames that trail cameras frequently capture.

=== 7.4.3 Object Detection with Custom Classes

**Custom object detection** combines classification with localization. Instead of labeling the entire image, you draw bounding boxes around specific objects and assign each box a class label. The trained model then detects and locates your custom object types within new images.

For the Trust, custom object detection can locate and classify multiple animals in a single frame. An image from a watering hole camera might contain two white-tailed deer and a red fox. Object detection returns three bounding boxes, each with its species label and confidence score. This is more informative than whole-image classification, which would only report the presence of those species without indicating where each animal appears or how many there are.

=== 7.4.4 Training Data Requirements

The quality and quantity of your training data directly determine your model's accuracy. Custom Vision has minimum requirements, but meeting the minimum produces mediocre results. Here are the practical guidelines:

* **Minimum images per class:** Custom Vision requires at least 5 images per class for classification and 15 for object detection. These minimums exist to allow training to start, not to produce a good model.
* **Recommended images per class:** Aim for 50 or more images per class to produce a model that generalizes well to new images.
* **Real-world variation:** Training images should reflect the conditions your model will encounter in production. For trail camera images, include examples from different times of day (dawn, midday, dusk, night with infrared), different weather conditions (clear, rainy, foggy, snowy), different distances (close-up, mid-range, far), and different camera angles. A model trained only on clear daylight images will struggle with dusk and infrared images.
* **Negative examples:** Include images that do not contain any of your target classes. Empty trail camera images, images of vegetation, and images of non-target species help the model learn what your classes are _not_.
* **Balanced classes:** Aim for roughly equal numbers of images across classes. A model trained on 200 deer images and 20 fox images will be biased toward predicting "deer."

=== 7.4.5 Training Cycle: Upload, Tag, Train, Evaluate, Iterate

The Custom Vision workflow follows a cycle:

1. **Upload** your images to a Custom Vision project through the portal, SDK, or REST API.
2. **Tag** each image with its class label (for classification) or draw bounding boxes and assign class labels (for object detection).
3. **Train** the model. Custom Vision offers quick training for rapid iteration and advanced training for better accuracy on larger datasets. Advanced training lets you specify a time budget in hours.
4. **Evaluate** the trained model using the built-in metrics (Section 7.4.6).
5. **Iterate** by analyzing misclassifications, adding more training images for weak classes, adjusting your class definitions, and retraining. Rarely does the first training iteration produce the best model.

This cycle mirrors the custom text classification workflow from Chapter 6. The fundamental principle is the same: train, evaluate, identify weaknesses, add data, retrain.

=== 7.4.6 Evaluation Metrics: Precision, Recall, mAP

Custom Vision reports three key metrics after each training iteration:

* **Precision** measures the percentage of predictions for a given class that were correct. If the model predicted "red fox" 100 times and 90 of those predictions were actually red foxes, precision for that class is 90%. High precision means few false positives.
* **Recall** measures the percentage of actual instances of a class that the model correctly identified. If there were 100 red fox images in the test set and the model correctly identified 80 of them, recall for that class is 80%. High recall means few false negatives.
* **Mean Average Precision (mAP)** is the primary metric for object detection models. It averages precision across all classes at various confidence thresholds, producing a single number that reflects overall detection quality.

These metrics often involve tradeoffs. Increasing your confidence threshold raises precision (fewer false positives) but lowers recall (more missed detections). For the Wildlife Trust, a rare-species detection system should favor recall -- missing a sighting of an endangered species is worse than a false alarm that a researcher can quickly dismiss.

=== 7.4.7 Deploying and Versioning

Once you have a trained model that meets your accuracy requirements, you **publish** it to a prediction endpoint. The published model receives a name (such as "species-classifier-v3") and becomes available for prediction requests through the REST API or SDK.

Custom Vision supports multiple published iterations simultaneously. This means you can publish a new model version without removing the previous one, allowing you to run them in parallel during a transition period. Your application can switch from the old endpoint to the new one with a configuration change rather than a code deployment.

Versioning is important for the Trust because their model will evolve over time. As cameras capture new species, as image quality changes with upgraded equipment, and as researchers identify misclassification patterns, the model will need retraining and republishing. Maintaining version history lets the team roll back to a previous model if a new version introduces unexpected regressions.

=== 7.4.8 Compact vs. Standard Models

Custom Vision offers two model domain types:

* **Standard models** run in the cloud and optimize for accuracy. They can be larger and more complex because they execute on Azure's infrastructure.
* **Compact models** are designed for export and edge deployment. They sacrifice some accuracy for smaller model size and faster inference, enabling them to run on mobile devices, IoT hardware, or Docker containers without a network connection.

For the Wildlife Trust, compact models open an intriguing possibility: deploying a species classifier directly on edge devices at trail camera stations, enabling real-time classification without transmitting every image back to the cloud. Only images flagged as containing rare or unusual species would be uploaded for researcher review, dramatically reducing bandwidth and storage costs.

[IMPORTANT]
====
Custom Vision requires a minimum of 5 images per class for classification and 15 for object detection, but these minimums produce weak models. Aim for 50 or more images per class with real-world variation -- different lighting, angles, distances, weather conditions, and backgrounds -- to produce meaningfully better results. A model trained on studio-quality images will fail on trail camera images captured at dusk in the rain.
====

== 7.5 Face Detection

Azure AI Vision includes a **Face** service that detects human faces in images and returns information about each detected face. For the Piedmont Wildlife Trust, face detection might seem irrelevant at first -- they are tracking animals, not people. But consider a related need: the Trust's visitor center uses cameras to count visitors entering the preserve, and the Trust wants to detect when unauthorized individuals appear on trail camera footage in restricted areas. Face detection is the service that makes these applications possible.

=== 7.5.1 What Face Detection Provides

Face detection identifies the location and attributes of human faces in an image. For each detected face, the service returns a **bounding box** showing where the face appears in the image. This basic detection capability works without any enrollment or training -- the service has a pre-built model that recognizes faces across a wide range of conditions, angles, and lighting.

Face detection answers the question "Are there human faces in this image, and where are they?" It does not answer the question "Whose face is this?" That distinction is critical, as you will see in Section 7.5.3.

=== 7.5.2 Available Attributes

Beyond the bounding box, the Face service can return additional attributes for each detected face:

* **Head pose** -- the tilt, roll, and yaw angles of the head, indicating which direction the person is looking.
* **Blur** -- whether the face is blurry and to what degree, which indicates image quality for that detection.
* **Exposure** -- whether the face is underexposed, well-exposed, or overexposed.
* **Noise** -- the level of visual noise in the face region.
* **Occlusion** -- whether parts of the face are blocked by objects, hair, or other obstructions.
* **Accessories** -- detection of glasses, headwear, and masks.
* **Quality for recognition** -- an overall quality assessment indicating whether the face image is suitable for further processing.

These attributes help you filter detections by quality. For the Trust's visitor counting application, you might accept all detected faces regardless of quality. For a security application comparing faces to a watchlist, you would filter for high-quality detections only.

=== 7.5.3 What Face Detection Does NOT Do (Identification Requires Approval)

Face detection tells you that a face exists in an image. **Face identification** tells you _whose_ face it is by matching it against a database of enrolled individuals. **Face verification** confirms whether two face images belong to the same person. These are fundamentally different capabilities with different ethical implications.

Azure's Face service technically supports identification and verification, but Microsoft gates access to these features behind an approval process. You cannot simply provision a Face resource and start identifying people. You must submit an application explaining your use case, and Microsoft reviews it against their Responsible AI principles before granting access.

This chapter covers face detection only. Identification and verification are out of scope for both this chapter and the AI-102 exam at the detection-only level.

=== 7.5.4 Responsible AI Considerations

Face detection and recognition technologies carry significant ethical weight. Bias in training data can cause higher error rates for certain demographic groups. Surveillance applications raise privacy concerns. Misidentification can have serious consequences in law enforcement and security contexts.

Microsoft has taken a deliberate stance on these issues:

* Fairness testing is built into the Face service to reduce demographic bias.
* Microsoft retired facial analysis features that attempted to infer emotional states, recognizing that these inferences are unreliable and potentially harmful.
* The gated access model for identification and verification forces organizations to articulate their use case and consider its ethical implications before deploying.

As a developer using Azure AI services, you share responsibility for ethical deployment. Building a visitor counter for a nature preserve is a low-risk application. Building a surveillance system that tracks individuals across a network of cameras is a high-risk application that requires careful ethical review -- even if you obtain Microsoft's approval.

=== 7.5.5 Microsoft's Requirements for Face-Related Capabilities

To use face identification, verification, or any capability that matches faces across images, you must:

1. Submit an intake form through Microsoft's Responsible AI application process.
2. Describe your specific use case and the population affected.
3. Confirm that your use case aligns with Microsoft's Responsible AI Standard.
4. Receive explicit approval before the features are enabled on your resource.

Detection-only features (locating faces, returning bounding boxes, and returning quality attributes) do not require this approval process and are available immediately when you provision a Face resource.

[WARNING]
====
Face recognition (identification and verification) requires Microsoft approval through a formal intake process. These features are not available by default. This chapter covers face detection only -- locating faces and returning attributes. If your application needs to identify specific individuals, you must apply for access and receive approval before development begins.
====

== 7.6 Vision-Enabled Generative AI

In Chapters 1 through 5, you worked with generative AI models that process text. You sent text prompts and received text responses. Modern generative AI models extend beyond text -- they can accept images as input and generate images as output. These **multimodal** capabilities blur the line between traditional vision services and generative AI, creating new possibilities for the Piedmont Wildlife Trust and any organization that works with visual content.

=== 7.6.1 Multimodal Models

A **multimodal model** processes multiple types of input -- text, images, audio, or video -- within a single model architecture. Azure OpenAI Service provides access to multimodal models (such as GPT-4o and GPT-4 Turbo with Vision) that accept both text and images in the same conversation. You can send an image alongside a text question and receive a text response that demonstrates visual understanding.

This is fundamentally different from the dedicated vision services covered in Sections 7.2 through 7.5. Those services return structured data: tags, bounding boxes, extracted text, face attributes. A multimodal model returns free-form text that can reason about the image, answer questions, describe complex scenes, and follow multi-turn conversations about visual content.

=== 7.6.2 Sending Images to Chat Completion

To send an image to a vision-enabled chat model through Azure OpenAI Service, you include the image in the messages array of your chat completion request. The image can be provided as a URL or as a base64-encoded string. The model processes the image alongside your text prompt and generates a response that incorporates visual understanding.

A researcher at the Piedmont Wildlife Trust could upload a trail camera photo and ask: "What species is in this image? Does it appear healthy? What is the approximate time of day based on the lighting?" The model would analyze the image and respond with a natural-language assessment -- something no dedicated Vision API can provide, because the question requires reasoning that combines visual perception with domain knowledge.

The key advantage of this approach is flexibility. You do not need to define categories in advance. You do not need training data. You can ask any question about any image, and the model will attempt an answer. The tradeoff is that the response is unstructured text rather than structured data, and accuracy depends on the model's general knowledge rather than domain-specific training.

=== 7.6.3 Use Cases: Visual Q&A, Troubleshooting, Accessibility

Vision-enabled generative AI excels in scenarios that require flexible reasoning:

* **Visual question answering (Q&A).** Users upload an image and ask open-ended questions. "What kind of vegetation is in the background?" "Is this animal a juvenile or an adult?" "What might be causing the discoloration on this plant?"
* **Troubleshooting.** A field researcher photographs a malfunctioning trail camera and asks "What might be wrong with this setup?" The model can identify loose connections, damaged components, or incorrect positioning.
* **Accessibility.** Vision-enabled models can generate rich, contextual descriptions of images for visually impaired users -- far more detailed and nuanced than the caption generation from Image Analysis.
* **Data validation.** A researcher can ask the model to verify whether an image's metadata matches its visual content: "This image is labeled as a coyote. Does that look correct?"

=== 7.6.4 Image Generation from Text Prompts (DALL-E)

Azure OpenAI Service also provides access to **DALL-E**, a model that generates images from text descriptions. You provide a text prompt describing the desired image, and DALL-E produces one or more images that match the description.

Image generation has practical applications beyond art and entertainment. The Wildlife Trust could use it to generate illustrations for educational materials ("Create an illustration showing the difference between a Cooper's hawk and a sharp-shinned hawk in flight"), create visual aids for volunteer training ("Generate an image showing correct trail camera mounting height and angle"), or produce placeholder images for reports when actual trail camera images are not available or suitable for publication.

DALL-E images are generated, not retrieved -- the model creates novel images rather than searching a database. This means the images will not be photorealistic representations of specific animals, but they can be useful for illustration, education, and conceptual visualization. Content filters automatically prevent the generation of harmful, deceptive, or inappropriate content.

=== 7.6.5 When to Use Vision-Enabled Chat vs. Dedicated Vision APIs

The choice between vision-enabled chat and dedicated Vision APIs depends on your task:

* **Use vision-enabled chat** when you need flexible, open-ended visual reasoning; when your questions vary from image to image; when you need natural-language explanations rather than structured data; or when you want multi-turn conversations about visual content.
* **Use dedicated Vision APIs** when you need structured, consistent output (tags, bounding boxes, extracted text); when you are processing images at scale and need high throughput; when you need deterministic results (the same image always produces the same tags); or when you need to meet specific latency requirements.

For the Wildlife Trust, the weekly processing of 50,000 trail camera images uses dedicated Vision APIs -- Image Analysis for tagging, Read API for text extraction, Custom Vision for species classification. But when a researcher encounters an unusual image and wants to discuss it interactively, vision-enabled chat provides the flexible reasoning that dedicated APIs cannot.

[TIP]
====
Use vision-enabled chat for flexible, conversational analysis where questions vary and natural-language explanations are valuable. Use dedicated Vision APIs for structured data extraction at scale where you need consistent output formats, high throughput, and predictable latency.
====

== 7.7 Comparing Vision Services: A Decision Framework

You have now learned about five distinct vision-related capabilities: Image Analysis, OCR/Read API, Custom Vision, Face Detection, and vision-enabled generative AI (including image generation). Each service has strengths, limitations, and ideal use cases. Choosing the right service for a given task is a critical skill -- and one that the AI-102 exam tests directly.

=== 7.7.1 Mapping Tasks to Services

The first step in choosing a service is clearly defining what you need to accomplish:

* **"What is in this image?"** -- Image Analysis (tags, objects, captions) for general categories; Custom Vision for domain-specific categories.
* **"What text appears in this image?"** -- Read API for raw text; Document Intelligence (Chapter 8) for structured form data.
* **"Are there faces in this image?"** -- Face Detection for locating faces; Face Identification (with approval) for identifying individuals.
* **"Answer this question about this image"** -- Vision-enabled chat for flexible, open-ended visual reasoning.
* **"Generate an image from this description"** -- DALL-E for text-to-image generation.

=== 7.7.2 Input Requirements and Output Formats

Each service has specific input requirements and produces different output formats:

* **Image Analysis** accepts JPEG, PNG, GIF, BMP, and TIFF images up to 20 MB. Output is structured JSON containing tags, objects with bounding boxes, captions, and smart crop regions.
* **Read API** accepts JPEG, PNG, BMP, TIFF, and PDF documents. Output is structured JSON containing pages, lines, words, and bounding polygons.
* **Custom Vision** accepts JPEG, PNG, GIF, and BMP images. Output is structured JSON containing predicted labels and confidence scores (classification) or bounding boxes with labels (detection).
* **Face Detection** accepts JPEG, PNG, GIF, and BMP images. Output is structured JSON containing face rectangles and optional attributes.
* **Vision-enabled chat** accepts JPEG, PNG, GIF, and WebP images as URLs or base64. Output is unstructured natural-language text.
* **DALL-E** accepts text prompts as input. Output is generated PNG images.

=== 7.7.3 Customization Spectrum

The services span a spectrum from fully pre-built to fully custom:

* **No customization needed:** Image Analysis, Read API, Face Detection. These work out of the box.
* **Full custom training:** Custom Vision. You provide training data and train your own model.
* **Prompt-based customization:** Vision-enabled chat. You customize behavior through system messages and prompt engineering rather than model training.

This spectrum mirrors the pre-built vs. custom pattern from Chapter 6. Start with pre-built, move to prompt-based customization, and invest in custom model training only when the first two options are insufficient.

=== 7.7.4 Latency and Throughput Profiles

Processing speed varies significantly across services:

* **Image Analysis** returns results in milliseconds, making it suitable for real-time applications.
* **Read API** is slightly slower, particularly for multi-page documents, but still returns results within seconds for single images.
* **Custom Vision** prediction latency is comparable to Image Analysis -- fast enough for real-time use.
* **Face Detection** returns results in milliseconds for images with a small number of faces.
* **Vision-enabled chat** has higher latency because it involves a large language model generating a response. It is best suited for interactive use rather than batch processing.
* **DALL-E** has the highest latency -- image generation can take several seconds per image.

For the Wildlife Trust's batch processing of 50,000 weekly images, latency differences compound. Processing each image through Image Analysis at 200ms per image takes about 2.8 hours. Processing each through vision-enabled chat at 5 seconds per image would take nearly 70 hours. This is why dedicated APIs are essential for batch workloads.

=== 7.7.5 Cost Considerations

Azure Vision services use different pricing models:

* **Image Analysis** and **Read API** charge per transaction (per image analyzed or per page read).
* **Custom Vision** charges for training time (per compute hour) and prediction transactions separately.
* **Face Detection** charges per transaction.
* **Vision-enabled chat** charges per token (input and output tokens, with image tokens calculated based on image size and resolution).
* **DALL-E** charges per generated image, with pricing varying by resolution.

For high-volume processing, the per-transaction cost of dedicated Vision APIs is typically much lower than the per-token cost of sending images through a large language model. The Wildlife Trust's 50,000 weekly images would cost a fraction through Image Analysis compared to vision-enabled chat -- another reason to reserve generative AI for interactive, ad-hoc analysis rather than batch processing.

== 7.10 Key Terms

image tagging:: An Image Analysis capability that returns descriptive words and phrases for an image, each with a confidence score indicating the model's certainty.

object detection:: A vision capability that identifies objects within an image and returns both a class label and a bounding box specifying each object's location.

bounding box:: A rectangle defined by coordinates (top-left corner, width, height) that outlines the location of a detected object within an image.

caption generation:: An Image Analysis capability that produces a natural-language sentence describing an image's content, using a model that understands both objects and their relationships.

dense captions:: Multiple region-specific captions generated for different areas of a single image, providing more detailed coverage than a single whole-image caption.

smart cropping:: A thumbnail generation technique that uses image analysis to identify and crop around the most important region of an image rather than applying a center crop.

confidence score:: A numeric value between 0 and 1 indicating how certain a model is about a prediction, used to filter results and trigger human review for uncertain cases.

Read API:: The Azure AI Vision service for extracting printed and handwritten text from images and documents, returning structured data including words, lines, and bounding polygons.

optical character recognition (OCR):: The technology for detecting and extracting text from images, converting visual text into machine-readable character data.

Custom Vision:: An Azure AI service that enables training domain-specific image classification and object detection models using your own labeled data.

multiclass classification:: A classification mode where each image receives exactly one label from a set of mutually exclusive categories.

multilabel classification:: A classification mode where each image can receive multiple labels simultaneously, allowing images to belong to more than one category.

mean average precision (mAP):: An evaluation metric for object detection that averages precision across all classes and confidence thresholds, providing a single measure of overall detection quality.

face detection:: A vision capability that identifies the presence and location of human faces in an image, returning bounding boxes and optional attributes without identifying specific individuals.

multimodal model:: A machine learning model that processes multiple types of input -- such as text, images, audio, or video -- within a single architecture, enabling cross-modal reasoning.

DALL-E:: An Azure OpenAI model that generates images from text descriptions, creating novel visual content based on natural-language prompts.
